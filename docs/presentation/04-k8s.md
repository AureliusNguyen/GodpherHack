# Kubernetes Deployment

Production-scale deployment for the Hub. Compose is enough for "one
team on one machine"; Kubernetes is the deployment shape when an
organization runs this for multiple teams or wants to survive node
failures.

Same `godpherhack/hub:dev` image. Different orchestrator, different
guarantees.

## File layout

```
deploy/k8s/
  namespace.yaml          # godpherhack namespace
  configmap.yaml          # non-secret config (HUB_BASE_URL, index name)
  secret.yaml.template    # template; secrets applied out of band
  deployment.yaml         # Hub pods (replicas, probes, security)
  service.yaml            # ClusterIP load-balancer
  ingress.yaml            # nginx-ingress with TLS + WS upgrade
  hpa.yaml                # HorizontalPodAutoscaler 2-10
  servicemonitor.yaml     # prometheus-operator scrape config (opt-in)
  kustomization.yaml      # bundles everything except secrets
  README.md               # apply order, scale notes, deferred items
```

## Why Kubernetes (not just Compose)

Five things K8s buys at once. Fault tolerance is one of them, not all.

| Capability | What it does for the Hub |
|---|---|
| **Horizontal scaling** | HPA watches CPU + memory across all Hub pods; scales replicas 2 to 10 when CPU > 70% or memory > 80%. Burst load (CTF starts at 8pm, 12 teammates connect) gets absorbed without manual intervention. |
| **Load balancing** | The Service distributes incoming requests across all healthy Hub pods. No single-pod bottleneck. |
| **Rolling updates** | Deploy a new Hub version with `kubectl set image deployment/godpherhack-hub hub=...`. K8s starts a new pod, waits for its readiness probe, drains traffic from the old one, kills it. Zero downtime to users. |
| **Self-healing** | Liveness probe at `/health` recycles wedged pods. Pod-level OOM/crash also triggers a fresh pod. |
| **Config + secret separation** | Pinecone key, JWT secret, GitHub OAuth creds live in a Secret that's mounted as env vars. Different env (dev/staging/prod) means different Secret, same image. |

## Manifest highlights

### `deployment.yaml` -- the Hub itself

- `replicas: 2` baseline (HPA scales up from there)
- Rolling update strategy: `maxUnavailable: 0, maxSurge: 1` -- always
  at least 2 pods serving traffic during an update
- Readiness probe: `/health` every 10s with 5s initial delay
- Liveness probe: `/health` every 30s with 15s initial delay
- Resource budget: requests 100m CPU / 128Mi RAM, limits 1000m / 512Mi
- Security context:
  - `runAsNonRoot: true`, `runAsUser: 1000`
  - `readOnlyRootFilesystem: true` (the Hub never writes to disk)
  - `allowPrivilegeEscalation: false`, all capabilities dropped
  - `seccompProfile: RuntimeDefault`
- Annotations for Prometheus pod-level scraping (fallback if the
  cluster doesn't have the operator's ServiceMonitor CRD installed)

### `hpa.yaml` -- the autoscaler

```yaml
minReplicas: 2
maxReplicas: 10
metrics:
  - cpu:    70% target utilization
  - memory: 80% target utilization
behavior:
  scaleUp:   30s stabilization window
  scaleDown: 5min stabilization window
```

Scale-up is aggressive (react to load fast); scale-down is patient
(don't thrash on transient dips).

### `ingress.yaml` -- TLS + WebSocket upgrade

```yaml
nginx.ingress.kubernetes.io/proxy-http-version: "1.1"
nginx.ingress.kubernetes.io/proxy-read-timeout: "3600"
```

Critical for the collab WebSocket at `/ws/collab`: nginx defaults to
HTTP/1.0, which kills the upgrade. Setting `proxy-http-version: 1.1`
+ a long read timeout keeps long-lived connections alive.

### `service.yaml`

ClusterIP on port 80, targeting container port `http` (3000). Internal
to the cluster; the Ingress is what's exposed publicly.

## What's deferred

From the README in `deploy/k8s/`:

- **Service mesh / mTLS between Hub instances** -- not needed at this
  scale; Istio/Linkerd adds operational overhead.
- **Network policies** -- the Hub only needs egress to GitHub
  (OAuth), Pinecone, and Anthropic. A real deploy should lock this
  down with a `NetworkPolicy`. Documented but not shipped.
- **PodDisruptionBudget** -- the rolling-update strategy already
  gives zero-downtime deploys; PDB is good hygiene during voluntary
  evictions but not blocking.
- **Multi-instance presence/feed sync** -- this is the big one.
  CollabHub is in-memory per pod, so users only see peers on the
  same pod. Sticky sessions via ingress affinity is a workaround;
  Redis Pub/Sub on the existing PresenceStore interface is the
  correct fix. Documented as the scale path.

## Validating without a real cluster

The manifests are reviewed via `kubectl apply --dry-run=client`. No
cluster is required for the demo:

```bash
kubectl apply --dry-run=client -k deploy/k8s/
```

Output (this machine):

```
namespace/godpherhack created (dry run)
configmap/godpherhack-hub-config created (dry run)
service/godpherhack-hub created (dry run)
deployment.apps/godpherhack-hub created (dry run)
horizontalpodautoscaler.autoscaling/godpherhack-hub created (dry run)
ingress.networking.k8s.io/godpherhack-hub created (dry run)
```

Six resources apply cleanly. ServiceMonitor is commented out in the
kustomization because it requires the prometheus-operator CRDs;
uncomment it when deploying into a kube-prometheus-stack cluster.

## Slide talking points

- Same image as Compose. Compose vs K8s is a deployment-scale
  decision, not a code change.
- Horizontal Pod Autoscaler: 2 minimum, 10 maximum. Variable load
  during a CTF event is exactly the shape HPA handles well.
- Hardened pod spec: non-root, read-only filesystem, dropped caps,
  seccomp default. The Hub never writes locally so read-only FS
  costs nothing and removes a class of attacks.
- Ingress with WebSocket upgrade was a deliberate config: collab
  uses long-lived `/ws/collab` connections, and nginx defaults
  would have killed them.
- The honest scale path is documented: in-memory presence works
  per-pod; multi-pod requires the Redis swap. Interface stays the
  same, only the store changes.
- Manifests validate via `kubectl --dry-run`; we did not need a
  live cluster to verify them. A real deploy is one `kubectl apply
  -k deploy/k8s/` away.
